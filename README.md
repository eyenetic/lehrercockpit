# Lehrercockpit

A multi-user SaaS teacher cockpit for Berlin schools. Each teacher gets their own individually configured dashboard with modules for WebUntis, itslearning, Nextcloud, grades, and more. An admin manages access and central data sources (Orgaplan, Klassenarbeitsplan) via an admin panel.

**Deployment:** Frontend on Netlify (static), backend on Render (Flask + gunicorn + PostgreSQL).

---

## Architecture Overview

```
Frontend (Netlify)               Backend (Render)             Database (PostgreSQL on Render)
app.lehrercockpit.com ─────────► api.lehrercockpit.com ─────► PostgreSQL
  index.html                       Flask + gunicorn
  login.html                       app.py (WSGI entry)
  admin.html                       /api/v2/* blueprints
  onboarding.html                  /api/* legacy endpoints
```

- **Frontend:** Vanilla HTML/CSS/JS — no build step, served by Flask
- **Backend:** Python 3.11, Flask, gunicorn — Dockerized on Render
- **Database:** PostgreSQL on Render, accessed via `psycopg` (psycopg3)

## Authentication Model

- **Access codes only** — no username/password login
- Each teacher receives a 32-character alphanumeric code generated by the admin
- Codes are stored only as **argon2id hashes** (`time_cost=3`, `memory_cost=65536`, `parallelism=1`)
- On login, the backend pre-filters candidates by the first 8 characters of the code (`code_prefix` column, O(1) index lookup), then verifies via argon2id — backward compatible with legacy codes (NULL prefix)
- Successful login sets an **HttpOnly session cookie** (`lc_session`, 30-day TTL)
- Cookie settings: `HttpOnly=True`, `Secure=True` in production, `SameSite=None` in production (required for cross-origin Netlify → Render), `SameSite=Lax` in local dev

## Module System

Modules are seeded into the `modules` table at startup and assigned to each user on creation.

| Module ID | Display Name | Type | Requires Config |
|---|---|---|---|
| `itslearning` | itslearning Lernplattform | `individual` | ✅ (URL, username, password) |
| `nextcloud` | Nextcloud | `individual` | ❌ |
| `webuntis` | WebUntis Stundenplan | `individual` | ✅ (iCal URL) |
| `orgaplan` | Orgaplan | `central` | ❌ (admin sets URL in system_settings) |
| `klassenarbeitsplan` | Klassenarbeitsplan | `central` | ❌ (admin sets URL in system_settings) |
| `noten` | Noten | `individual` | ❌ |
| `mail` | Dienstmail (lokal) | `local` | ❌ |

**Module types:**
- **`individual`** — Each teacher configures their own credentials (stored in `user_module_configs`)
- **`central`** — Admin configures a shared URL in `system_settings`; all teachers see the same data
- **`local`** — Only usable on a local dev instance (e.g. Apple Mail/IMAP); not suitable for SaaS

## Admin Workflow

### Bootstrap (first deploy)

On first startup with an empty database and `DATABASE_URL` set, the backend automatically creates a bootstrap admin and prints the access code to stdout (Render logs):

```
============================================================
[BOOTSTRAP] Erster Admin-User wurde erstellt.
[BOOTSTRAP] Zugangscode: <32-char-code>
[BOOTSTRAP] User-ID: 1 | Name: Bootstrap Admin
[BOOTSTRAP] Bitte diesen Code sofort sichern und im
[BOOTSTRAP] Admin-Panel den Namen anpassen.
============================================================
```

This is idempotent — it does nothing if any admin already exists.

### Creating Teachers

1. Log in as admin → visit `/admin`
2. Tab **Users** → "Neue Lehrkraft anlegen" → enter first name, last name, role
3. The generated access code is displayed **once** — copy it and deliver it securely
4. The code is stored only as an argon2id hash — the plaintext is never saved

### Regenerating Codes

Admin panel → teacher row → "Code erneuern". The old code is immediately invalidated.

### System Settings (Central Modules)

Admin panel → **Settings** tab:
- `orgaplan_url` / `orgaplan_pdf_url` — URL for the Orgaplan PDF
- `klassenarbeitsplan_url` — URL for the Klassenarbeitsplan
- Any other key-value pairs stored as JSONB

## Teacher Onboarding

1. Teacher visits `/login` and enters their access code
2. Dashboard checks `GET /api/v2/dashboard/onboarding-status` — if modules with `requires_config=true` are unconfigured, `needs_onboarding: true`
3. Teacher is redirected to `/onboarding.html`
4. Steps: select visible modules → enter credentials for required modules (itslearning URL/username/password, WebUntis iCal URL, etc.)
5. `POST /api/v2/dashboard/onboarding/complete` marks onboarding done
6. Redirect to `/` (dashboard)

## Dashboard & Layout

- `GET /api/v2/dashboard` — aggregated payload: user info + module list + onboarding status
- `GET /api/v2/dashboard/layout` — module visibility and sort order for the logged-in user
- `PUT /api/v2/dashboard/layout` — save updated module order/visibility
- `PUT /api/v2/dashboard/layout/<module_id>/visibility` — toggle single module visibility
- Drag-to-reorder: implemented via number inputs (not HTML5 drag-and-drop)
- Module visibility and order are stored per-user in `user_modules`

> **Note:** `src/app.js` currently still calls the legacy `GET /api/dashboard` (v1) endpoint for the main dashboard payload alongside the v2 modular API. Full migration to v2 is a planned next step.

## Local Development

The app runs without a build step:

```bash
pip install -r requirements.txt

# Option 1: Legacy single-user server (no multi-user, no DB)
python server.py
# → http://localhost:8080

# Option 2: Flask (multi-user, requires DATABASE_URL or falls back to file store)
flask run --port 8080
# or:
gunicorn app:app --bind 0.0.0.0:8080 --workers 1
```

Copy `.env.local` for local settings:

```bash
cp config/mail.env.example .env.local
# Edit .env.local — it is gitignored
```

> **Note:** `server.py` (stdlib `ThreadingHTTPServer`) does not support Flask blueprints and cannot serve v2 API endpoints. Use `flask run` or `gunicorn` for multi-user local development.

## Security

| Mechanism | Implementation |
|---|---|
| Access codes | argon2id hashed (`time_cost=3, memory_cost=64MB`), never stored plaintext |
| Module credentials | Fernet-encrypted at rest when `ENCRYPTION_KEY` is set; stored as `enc:<ciphertext>` in JSONB; backward compatible with existing plaintext records |
| API responses | Sensitive fields (containing `password`, `secret`, `token`, `credential` in the key name) masked as `***` in all `/api/v2/modules/*/config` and `/api/v2/dashboard/module-config/*` responses |
| Sessions | `SameSite=None; Secure=True` cookies in production (required for cross-origin Netlify → Render requests) |
| Bootstrap | PostgreSQL advisory-locked, one-time, logs access code to stdout (Render logs) |

## Known Limitations

- **v1 dashboard API still used for main payload:** `src/app.js` calls the legacy `GET /api/dashboard` endpoint for the main dashboard payload (webuntis, itslearning, orgaplan, classwork). Grades/notes now use v2 endpoints. Full migration to per-module v2 data endpoints is a planned next step.
- **DB tests require `TEST_DATABASE_URL`:** Without it, all DB-dependent tests are skipped locally. CI uses a PostgreSQL 15 service container — no secret needed in CI.
- **Credentials stored as plaintext if `ENCRYPTION_KEY` is not set** — set `ENCRYPTION_KEY` on Render in production.
- **Existing plaintext values in DB must be backfilled** after `ENCRYPTION_KEY` is set: run `scripts/backfill_encryption.py` (see Scripts & Maintenance section).
- **`mail` module is local-only:** The `mail` module (`local` type) requires direct Apple Mail/IMAP access on the local machine. It is not a SaaS module and is hidden by default (`default_visible=FALSE`).
- **No email-based password reset:** By design — access codes are the auth mechanism. Code rotation is admin-initiated via the admin panel.
- **Drag-to-reorder uses number inputs:** The dashboard module reordering UI uses numeric order inputs, not HTML5 drag-and-drop.
- **`code_prefix` backfill not possible for existing codes:** argon2id is one-way; old codes can only get a prefix after rotation. Existing NULL-prefix codes still work (fallback to full-scan).

## Scripts & Maintenance

### Encrypt existing module configs

If `ENCRYPTION_KEY` was set after teachers had already saved module credentials, existing records remain plaintext. Run the backfill script to encrypt them:

```bash
ENCRYPTION_KEY=<your-key> DATABASE_URL=<url> python scripts/backfill_encryption.py --dry-run
# Review output, then run without --dry-run to apply:
ENCRYPTION_KEY=<your-key> DATABASE_URL=<url> python scripts/backfill_encryption.py
```

The script is idempotent — it is safe to run multiple times.
